Setup

Graph: \(C_4\) — cycle on 4 nodes \(\{0,1,2,3\}\), edges \(\{(0,1),(1,2),(2,3),(3,0)\}\)

Node features: \(\mathbf{x} = [2, 1, 1, 1]^\top\)

The number of nodes: \(N = 4\).

\(\mathbf{X} \in \mathbb{R}^{N \times F}\) is the node feature matrix, where row \(i\) is the feature vector of node \(i\).

Scalar features, \(F=1\):

\[\mathbf{X} = \begin{bmatrix} 2 \\\\ 1 \\\\ 1 \\\\ 1 \end{bmatrix} \in \mathbb{R}^{4 \times 1}\]

Phase 0: Foundations

Phase 0.1 - Concatenation

You know matrix multiplication joins matrices through inner dimensions. Concatenation is simpler — it’s just physically stacking values with no arithmetic.

Given two vectors:

\[\mathbf{a} = \begin{bmatrix} 2 \end{bmatrix}, \quad \mathbf{b} = \begin{bmatrix} 1 \end{bmatrix}\]

Their concatenation is:

\[[\mathbf{a} \| \mathbf{b}] = \begin{bmatrix} 2 \\\\ 1 \end{bmatrix}\]

No multiplication, no addition. Just placing them side by side into one longer vector. The \(\|\) symbol means exactly that — join.

In our \(C_4\) example, node 0 has feature \(h_0 = 2\) and node 1 has feature \(h_1 = 1\). Before computing the message on edge \(0 \to 1\), we first form the input to the MLP:

\[[h_0 \| h_1] = \begin{bmatrix} 2 \\\\ 1 \end{bmatrix} = \mathbf{m}_{01}\]

This is a 2-dimensional vector assembled from two 1-dimensional scalars. If nodes had, say, 4 features each, the concatenation would produce an 8-dimensional vector. The dimension simply adds.

\[\dim([\mathbf{a} \| \mathbf{b}]) = \dim(\mathbf{a}) + \dim(\mathbf{b})\]

That’s the entirety of concatenation.

Phase 0.2 - Broadcasting

Broadcasting answers one question: what do you do when an operation requires two tensors of matching shape, but they don’t match?

Let’s build the need for it concretely first.

1. The situation

After computing messages on all edges, you have a message matrix — one row per edge:

\[\text{messages} = \begin{bmatrix} m_{01} \\\\ m_{10} \\\\ m_{12} \\\\ m_{21} \\\\ m_{23} \\\\ m_{32} \\\\ m_{30} \\\\ m_{03} \end{bmatrix} \quad \text{shape: } [8 \times H]\]

where \(H\) is the hidden dimension (how many numbers represent each message). Say \(H = 4\) for concreteness.

Each message row needs to be sent to its destination node. The destination indices for these 8 edges are:

\[\text{dst} = \begin{bmatrix} 1 \\\\ 0 \\\\ 2 \\\\ 1 \\\\ 3 \\\\ 2 \\\\ 0 \\\\ 3 \end{bmatrix} \quad \text{shape: } [8]\]

This says: message 0 goes to node 1, message 1 goes to node 0, message 2 goes to node 2, and so on.


2. The mismatch

To tell scatter_add_ where each message goes, it needs dst and messages to have the same shape \([8 \times 4]\).

But dst is shape \([8]\) — just a list of 8 numbers. messages is \([8 \times 4]\).

They don’t match. Broadcasting fixes this.


3. What broadcasting does

It repeats the smaller tensor along the missing dimension until shapes align:

\[\text{dst} = \begin{bmatrix} 1 \\\\ 0 \\\\ 2 \\\\ 1 \\\\ 3 \\\\ 2 \\\\ 0 \\\\ 3 \end{bmatrix} \xrightarrow{\text{expand to } [8 \times 4]} \begin{bmatrix} 1 & 1 & 1 & 1 \\\\ 0 & 0 & 0 & 0 \\\\ 2 & 2 & 2 & 2 \\\\ 1 & 1 & 1 & 1 \\\\ 3 & 3 & 3 & 3 \\\\ 2 & 2 & 2 & 2 \\\\ 0 & 0 & 0 & 0 \\\\ 3 & 3 & 3 & 3 \end{bmatrix}\]

No new data is created. Each destination index is simply copied across all \(H\) columns — because the same index applies to every dimension of that message row.

This is what dst.unsqueeze(-1).expand_as(messages) does in code: - unsqueeze(-1) reshapes \([8]\) → \([8 \times 1]\) - expand_as(messages) stretches \([8 \times 1]\) → \([8 \times 4]\) by repeating

The core rule

A dimension of size 1 can be stretched to match any size — because repeating one value across positions costs no new information.


Revision

That’s actually not quite right — let me show you why before we move on.

unsqueeze(-1) does one thing only: it inserts a new dimension of size 1 at the last position.

So the transformation is:

\[[6] \xrightarrow{\text{unsqueeze(-1)}} [6 \times 1]\]

The result is — \([6 \times 1]\).

The expansion to \([6 \times 3]\) happens after, in the separate expand_as step. These are two distinct operations:

Operation Input shape Output shape
unsqueeze(-1) \([6]\) \([6 \times 1]\)
expand_as(messages) \([6 \times 1]\) \([6 \times 3]\)

The reason they’re separated: unsqueeze tells PyTorch where the new dimension goes, and expand_as tells it how far to stretch it. One step cannot do both.

Revisiting the original \(C_4\) case with \(H=4\):

\[[8] \xrightarrow{\text{unsqueeze(-1)}} [8 \times 1] \xrightarrow{\text{expand_as}} [8 \times 4]\]

Phase 0.3 - MLP Mathematics

MLP is actually a strict subset of what a CNN does.

1. What you already know from CNN

Based on the concept of CNN, a convolutional layer does this to a spatial input:

\[\mathbf{y} = \text{ReLU}(\mathbf{W} * \mathbf{x} + \mathbf{b})\]

where \(*\) is the convolution — a sliding dot product across spatial positions.

Now ask: what if the input had no spatial dimensions at all — just a single vector?

The sliding window collapses to a single position. The convolution reduces to a plain dot product. That is an MLP layer.


2. One MLP layer

\[\mathbf{y} = \text{ReLU}(\mathbf{W}\mathbf{x} + \mathbf{b})\]

Symbol Meaning
\(\mathbf{x} \in \mathbb{R}^{d_{in}}\) input vector
\(\mathbf{W} \in \mathbb{R}^{d_{out} \times d_{in}}\) learned weight matrix
\(\mathbf{b} \in \mathbb{R}^{d_{out}}\) learned bias vector
\(\mathbf{y} \in \mathbb{R}^{d_{out}}\) output vector

ReLU is applied elementwise: \(\text{ReLU}(z) = \max(0, z)\).

Stack two such layers and you have a 2-layer MLP — exactly what \(\text{MLP}_\text{msg}\) and \(\text{MLP}_\text{update}\) are.


3. Concretely on edge \(0 \to 1\)

Input after concatenation: \(\mathbf{x} = [2, 1]^\top\), so \(d_{in} = 2\). Let \(d_{out} = H = 2\) for simplicity, with made-up weights:

\[\mathbf{W} = \begin{bmatrix} 1 & 0 \\\\ 0 & 1 \end{bmatrix}, \quad \mathbf{b} = \begin{bmatrix} 0 \\\\ 0 \end{bmatrix}\]

Then:

\[\mathbf{m}_{01} = \text{ReLU}\left(\begin{bmatrix} 1 & 0 \\\\ 0 & 1 \end{bmatrix} \begin{bmatrix} 2 \\\\ 1 \end{bmatrix} + \begin{bmatrix} 0 \\\\ 0 \end{bmatrix}\right) = \text{ReLU}\left(\begin{bmatrix} 2 \\\\ 1 \end{bmatrix}\right) = \begin{bmatrix} 2 \\\\ 1 \end{bmatrix}\]

That vector \([2, 1]^\top\) is the message on edge \(0 \to 1\).

Phase 0.4 - scatter_add_ (Index-Based Accumulation)

1. The problem it solves

After computing all 8 messages on \(C_4\), you need to sum them into their destination nodes. Node 0 should receive messages from nodes 1 and 3. Node 1 should receive from nodes 0 and 2. And so on.

You have:

\[\text{messages} = \begin{bmatrix} \mathbf{m}_{01} \\\\ \mathbf{m}_{10} \\\\ \mathbf{m}_{12} \\\\ \mathbf{m}_{21} \\\\ \mathbf{m}_{23} \\\\ \mathbf{m}_{32} \\\\ \mathbf{m}_{30} \\\\ \mathbf{m}_{03} \end{bmatrix}, \quad \text{dst} = \begin{bmatrix} 1 \\\\ 0 \\\\ 2 \\\\ 1 \\\\ 3 \\\\ 2 \\\\ 0 \\\\ 3 \end{bmatrix}\]

scatter_add_ reads dst row by row and adds each message into the corresponding row of an accumulator matrix agg of shape \([N \times H] = [4 \times 2]\), initialized to zero.


2. Step-by-step on \(C_4\)

\[\text{agg} = \begin{bmatrix} 0 & 0 \\\\ 0 & 0 \\\\ 0 & 0 \\\\ 0 & 0 \end{bmatrix}\]

Step Message dst Operation
1 \(\mathbf{m}_{01} = [2,1]\) 1 agg[1] += [2,1]
2 \(\mathbf{m}_{10} = [1,2]\) 0 agg[0] += [1,2]
3 \(\mathbf{m}_{12} = [1,1]\) 2 agg[2] += [1,1]
4 \(\mathbf{m}_{21} = [1,1]\) 1 agg[1] += [1,1]
5 \(\mathbf{m}_{23} = [1,1]\) 3 agg[3] += [1,1]
6 \(\mathbf{m}_{32} = [1,1]\) 2 agg[2] += [1,1]
7 \(\mathbf{m}_{30} = [1,2]\) 0 agg[0] += [1,2]
8 \(\mathbf{m}_{03} = [2,1]\) 3 agg[3] += [2,1]

Final result:

\[\text{agg} = \begin{bmatrix} 2 & 4 \\\\ 3 & 2 \\\\ 2 & 2 \\\\ 3 & 2 \end{bmatrix}\]

Row \(i\) of agg is exactly \(\mathbf{M}_i\) — the aggregated neighborhood summary for node \(i\).


3. How each operation built the final agg

agg starts as a \(4 \times 2\) zero matrix — one row per node:

\[\text{agg} = \begin{bmatrix} 0 & 0 \\\\ 0 & 0 \\\\ 0 & 0 \\\\ 0 & 0 \end{bmatrix} \leftarrow \text{node 0, 1, 2, 3}\]

scatter_add_ processes messages one at a time. Each message has a destination index — that index tells it which row to add into. It’s a running sum, not a replacement.

Tracing only node 0’s row to keep it tight:

\[\text{agg}[0] = \underbrace{[0,0]}_{\text{init}} \xrightarrow{+\,\mathbf{m}_{10}=[1,2]} [1,2] \xrightarrow{+\,\mathbf{m}_{30}=[1,2]} [2,4]\]

Two messages landed on row 0, both identical because \(h_1 = h_3\). Their sum is \([2,4]\).

The same accumulation runs independently for every other row — messages destined for node 1 pile into row 1, for node 2 into row 2, and so on. No row interferes with another.

Full final state:

\[\text{agg} = \begin{bmatrix} 2 & 4 \\\\ 3 & 2 \\\\ 2 & 2 \\\\ 3 & 2 \end{bmatrix} \leftarrow \mathbf{M}_0,\ \mathbf{M}_1,\ \mathbf{M}_2,\ \mathbf{M}_3\]

Phase : \(C_4\) as a Data Structure

Phase 1.1 - Nodes and Feature Matrix

\(C_4\) is a cycle graph with 4 nodes and 4 edges. You may imagine its structure. The question is: how does a computer represent it?

Nodes are represented as rows in a feature matrix \(\mathbf{H}\):

\[\mathbf{H} = \begin{bmatrix} 2 \\\\ 1 \\\\ 1 \\\\ 1 \end{bmatrix} \quad \text{shape: } [4 \times 1]\]

Row \(i\) is the feature vector of node \(i\). Here each node has a single scalar feature, so \(F = 1\) and the matrix is \([N \times F] = [4 \times 1]\).

In general, if each node carried a 16-dimensional feature vector, \(\mathbf{H}\) would be \([4 \times 16]\), i.e., 4 nodes and 16 features. The structure is the same–rows are nodes, columns are features.

Phase 1.2 - Edge Index Representation

Edges are stored as a pair of index lists — source and destination:

\[\text{src} = \begin{bmatrix} 0 \\\\ 1 \\\\ 1 \\\\ 2 \\\\ 2 \\\\ 3 \\\\ 3 \\\\ 0 \end{bmatrix}, \quad \text{dst} = \begin{bmatrix} 1 \\\\ 0 \\\\ 2 \\\\ 1 \\\\ 3 \\\\ 2 \\\\ 0 \\\\ 3 \end{bmatrix}\]

Row \(k\) of both together describes one directed edge: \(\text{src}[k] \to \text{dst}[k]\). \(C_4\) has 4 undirected edges, each stored twice — once per direction — giving \(2E = 8\) rows.

Visually:

\[0 \leftrightarrow 1 \leftrightarrow 2 \leftrightarrow 3 \leftrightarrow 0\]

Every node has exactly 2 neighbors. That means every node will receive exactly 2 messages per round — which you already saw in the scatter_add_ table.


Question: If \(C_4\) had edge features \(\mathbf{e}_{ij}\) of dimension \(F_e = 3\), what would the shape of the edge feature matrix be?

Not quite — let’s think through it carefully.

The edge feature matrix stores one feature vector per edge. How many edges does \(C_4\) have in its directed representation?

4 undirected edges, each stored twice, giving \(2E = 8\) directed edges.

So the matrix needs 8 rows — one per directed edge — and 3 columns for the \(F_e = 3\) features.

That makes the answer : \([8 \times 3]\).

Anything indexed by nodes has \(N\) rows. Anything indexed by edges has \(2E\) rows.


Question: The message matrix (output of \(\text{MLP}_\text{msg}\)) lives in which category — node-indexed or edge-indexed — and what is its shape on \(C_4\) with \(H = 4\)?

The MLP\(_\text{msg}\) takes one concatenated input per edge and produces one output per edge. So is the output indexed by nodes or by edges?

And separately:

On \(C_4\) with \(2E = 8\) directed edges and hidden dimension \(H = 4\), how many rows must the message matrix have?

Using those two observations — we get \([8 \times 4]\) shape.


Revision

You now have the full data structure picture:

Object Shape Indexed by
\(\mathbf{H}\) \([4 \times 1]\) nodes
src, dst \([8]\) directed edges
messages \([8 \times H]\) directed edges
agg \(\mathbf{M}\) \([4 \times H]\) nodes

Every shape in the full data flow reduces to one of these two categories — node-indexed or edge-indexed. Keep that table in mind as we proceed.

Phase 2 - Message Computation

Phase 2.1 - What a message is & why it’s formed this way?

From CNN — a convolutional filter looks at a local neighborhood of pixels to extract spatial patterns. Message passing is the graph analogue: each edge carries information from its local neighborhood — the two endpoint nodes — into a learned representation.

Formally, for every directed edge \(i \to j\):

\[\mathbf{m}_{ij} = \text{MLP}_\text{msg}([\mathbf{h}_i \| \mathbf{h}_j])\]

The input is the concatenation of both endpoints. Why both? Because the message should capture not just who is sending, but the relationship between sender and receiver. This is strictly richer than classic GCN which only uses \(\mathbf{h}_i\).

Phase 2.2 - Computing \(\mathbf{m}_{01}\) and \(\mathbf{m}_{03}\)

For \(C_4\) with \(\mathbf{h} = [2,1,1,1]^\top\) and the same identity weights from Phase 0.3:

\[\mathbf{W} = \begin{bmatrix} 1 & 0 \\\\ 0 & 1 \end{bmatrix}, \quad \mathbf{b} = \begin{bmatrix} 0 \\\\ 0 \end{bmatrix}\]

Edge \(0 \to 1\):

\[\mathbf{m}_{01} = \text{ReLU}\left(\mathbf{W}[h_0 \| h_1] + \mathbf{b}\right) = \text{ReLU}\left(\begin{bmatrix}1&0 \\\\ 0&1\end{bmatrix}\begin{bmatrix}2 \\\\ 1\end{bmatrix}\right) = \begin{bmatrix}2 \\\\ 1\end{bmatrix}\]

Edge \(0 \to 3\):

\[\mathbf{m}_{03} = \text{ReLU}\left(\mathbf{W}[h_0 \| h_3] + \mathbf{b}\right) = \text{ReLU}\left(\begin{bmatrix}1&0 \\\\ 0&1\end{bmatrix}\begin{bmatrix}2 \\\\ 1\end{bmatrix}\right) = \begin{bmatrix}2 \\\\ 1\end{bmatrix}\]

Identical inputs → identical outputs. This is not a coincidence — it is a structural fact about \(C_4\).


1. The WL Limit — First Encounter

Node 0 sends the same message to both its neighbors because \(h_1 = h_3 = 1\). Now flip the question:

From node 0’s perspective — can it tell nodes 1 and 3 apart based on what it sends them?

It cannot. And more critically — when node 0 receives messages back from nodes 1 and 3 in the next phase, those will also be identical. Node 0 cannot distinguish its two neighbors.

This is the Weisfeiler-Lehman expressiveness limit: if two nodes have identical neighborhood structure and features, no standard message passing GNN can distinguish them. We’ll see this collapse explicitly in Phase 3.

Phase 3 - Aggregation

Phase 3.1 - scatter_add_ on \(C_4\)

You’ve already seen this mechanics in Phase 0.4. Now we run it with real messages from Phase 2.

First, the full message matrix. Using identity weights, each message \(\mathbf{m}_{ij} = [h_i, h_j]^\top\):

Edge src → dst Message \((m_{src \to dst})\)
\(0\to1\) 0 → 1 \([2,1]\)
\(1\to0\) 1 → 0 \([1,2]\)
\(1\to2\) 1 → 2 \([1,1]\)
\(2\to1\) 2 → 1 \([1,1]\)
\(2\to3\) 2 → 3 \([1,1]\)
\(3\to2\) 3 → 2 \([1,1]\)
\(3\to0\) 3 → 0 \([1,2]\)
\(0\to3\) 0 → 3 \([2,1]\)

Accumulating into agg by destination:

\[\mathbf{M}_0 = \mathbf{m}_{10} + \mathbf{m}_{30} = [1,2]+[1,2] = [2,4]\] \[\mathbf{M}_1 = \mathbf{m}_{01} + \mathbf{m}_{21} = [2,1]+[1,1] = [3,2]\] \[\mathbf{M}_2 = \mathbf{m}_{12} + \mathbf{m}_{32} = [1,1]+[1,1] = [2,2]\] \[\mathbf{M}_3 = \mathbf{m}_{23} + \mathbf{m}_{03} = [1,1]+[2,1] = [3,2]\]

\[\text{agg} = \begin{bmatrix} 2&4 \\\\ 3&2 \\\\ 2&2 \\\\ 3&2 \end{bmatrix}\]

Phase 3.2 - The WL Collapse

Look at rows 1 and 3:

\[\mathbf{M}_1 = \mathbf{M}_3 = [3,2]\]

After one full round of message passing, nodes 1 and 3 have identical aggregated summaries. Any GNN update applied to these identical \(\mathbf{M}\) values will produce identical updated features — the network provably cannot distinguish them.

This isn’t a weight-tuning problem. No training fixes it. It’s a structural limitation — the neighborhood of node 1 and node 3 look identical from the graph’s perspective.

Phase 4 - Node Update

Phase 4.1 - Why concatenate \(\mathbf{h}_i\) before updating?

Before the math — think about what \(\mathbf{M}_i\) alone contains.

\(\mathbf{M}_0 = [2,4]\) summarizes node 0’s neighborhood. But it contains no information about node 0 itself. If you replaced \(\mathbf{h}_0\) with \(\mathbf{M}_0\) directly, node 0 would forget it ever had feature \(h_0 = 2\).

The fix is to concatenate before updating:

\[\mathbf{h}_i' = \text{MLP}_\text{update}([\mathbf{h}_i \| \mathbf{M}_i])\]

The node retains its own identity alongside its neighborhood summary. The update MLP then learns how to blend the two.

Phase 4.2 - Computing \(\mathbf{h}_0'\)

For node 0, using the same identity weights and \(H=2\), but now \(d_{in} = 3\) because \([\mathbf{h}_0 \| \mathbf{M}_0] = [2 \| 2, 4] = [2, 2, 4]^\top\):

\[\mathbf{W}_\text{update} = \begin{bmatrix} 1&0&0 \\\\ 0&1&0 \end{bmatrix}, \quad \mathbf{b} = \begin{bmatrix}0 \\\\ 0\end{bmatrix}\]

\[\mathbf{h}_0' = \text{ReLU}\left(\begin{bmatrix}1&0&0 \\\\ 0&1&0\end{bmatrix}\begin{bmatrix}2 \\\\ 2 \\\\ 4\end{bmatrix}\right) = \text{ReLU}\left(\begin{bmatrix}2 \\\\ 2\end{bmatrix}\right) = \begin{bmatrix}2 \\\\ 2\end{bmatrix}\]

Phase 4.3 - All Four Updated Nodes

Applying the same procedure to every node:

Node \(\mathbf{h}_i\) \(\mathbf{M}_i\) \([\mathbf{h}_i \Vert \mathbf{M}_i]\) \(\mathbf{h}_i'\)
0 \([2]\) \([2,4]\) \([2,2,4]\) \([2,2]\)
1 \([1]\) \([3,2]\) \([1,3,2]\) \([1,3]\)
2 \([1]\) \([2,2]\) \([1,2,2]\) \([1,2]\)
3 \([1]\) \([3,2]\) \([1,3,2]\) \([1,3]\)

Notice nodes 1 and 3 produce identical \(\mathbf{h}'\) — the WL collapse persists through the update, exactly as predicted.


Question: Node 2 received from nodes 1 and 3, both with \(h=1\). Why is \(\mathbf{h}_2'\) different from \(\mathbf{h}_1'\) despite all three nodes starting with \(h=1\)?

Because, node 2’s neighbors both have \(h=1\), while node 1’s neighbors have \(h=2\) and \(h=1\)


On \(\mathbf{W}_\text{update}\)

\(\mathbf{W}_\text{update}\) — was assummed to be such and chosen deliberately to keep the arithmetic transparent. In a real GNN, \(\mathbf{W}_\text{update}\) is learned during training via backpropagation, just like CNN filter weights.

The \([2 \times 3]\) shape comes directly from the dimensions of the input and output:

\[\mathbf{W}_\text{update} \in \mathbb{R}^{d_{out} \times d_{in}}\]

Here:

  • \(d_{in} = 3\) because \([\mathbf{h}_i \| \mathbf{M}_i]\) concatenates a scalar \(\mathbf{h}_i \in \mathbb{R}^1\) with \(\mathbf{M}_i \in \mathbb{R}^2\), giving a 3-dimensional input
  • \(d_{out} = 2\) because we chose hidden dimension \(H = 2\)

So the matrix must be \([2 \times 3]\) to satisfy the matrix multiplication \(\mathbf{W}_\text{update} \cdot [\mathbf{h}_i \| \mathbf{M}_i]\):

\[[2 \times 3] \cdot [3 \times 1] = [2 \times 1]\]

The general rule — consistent with everything you’ve seen so far:

\[\mathbf{W} \in \mathbb{R}^{d_{out} \times d_{in}}, \quad d_{in} = F + H, \quad d_{out} = H\]

Phase 5 - Full Data Flow

Phase 5.1 - The Complete Shape Trace

Every transformation in one pipeline, using \(C_4\) values throughout:

\[[N, F] \xrightarrow{\text{gather}} [2E,\ 2F] \xrightarrow{\text{MLP}_\text{msg}} [2E, H] \xrightarrow{\text{scatter_add}} [N, H] \xrightarrow{\text{MLP}_\text{update}} [N, H]\]

Substituting \(N=4,\ F=1,\ E=4,\ H=2\):

\[[4,1] \xrightarrow{\text{gather}} [8,2] \xrightarrow{\text{MLP}_\text{msg}} [8,2] \xrightarrow{\text{scatter_add}} [4,2] \xrightarrow{\text{MLP}_\text{update}} [4,2]\]

Each arrow is one operation. Let’s nail what happens at each boundary.


Step 1 — Gather: \([4,1] \to [8,2]\)

For every directed edge, look up the features of its source and destination node from \(\mathbf{H}\), then concatenate them:

\[\text{edge } k: \quad [\mathbf{h}_{\text{src}[k]} \| \mathbf{h}_{\text{dst}[k]}]\]

8 edges × 2 concatenated scalars = \([8 \times 2]\). No arithmetic yet — pure lookup and concatenation.


Step 2 — \(\text{MLP}_\text{msg}\): \([8,2] \to [8,2]\)

Each of the 8 rows passes independently through the same MLP. Shape stays \([8 \times 2]\) because \(d_{in} = 2F = 2\) and \(d_{out} = H = 2\).


Step 3 — scatter_add_: \([8,2] \to [4,2]\)

8 edge-indexed messages collapse into 4 node-indexed aggregations. Edge dimension \(\to\) node dimension. This is the only step that reduces the number of rows.


Step 4 — \(\text{MLP}_\text{update}\): \([4,2] \to [4,2]\)

Each node concatenates its own feature \(\mathbf{h}_i \in \mathbb{R}^1\) with \(\mathbf{M}_i \in \mathbb{R}^2\) internally, giving \(d_{in} = 3\), \(d_{out} = H = 2\). The 4 nodes pass independently through the same MLP. Output is the new feature matrix \(\mathbf{H}' \in [4 \times 2]\).

Phase 5.2 - What did one round actually accomplish?

Before message passing, every node knew only itself:

\[\mathbf{h}_0 = 2, \quad \mathbf{h}_1 = \mathbf{h}_2 = \mathbf{h}_3 = 1\]

After one round, every node knows itself plus a summary of its immediate neighbors:

\[\mathbf{h}_0' = [2,2], \quad \mathbf{h}_1' = \mathbf{h}_3' = [1,3], \quad \mathbf{h}_2' = [1,2]\]

Each updated feature vector encodes a 1-hop neighborhood summary. Stack another round on top, and every node would encode a 2-hop summary. This is how GNNs build up global graph awareness — one hop at a time.

Question: The table shows node 0 on \(C_4\) sees nodes 1 and 3 after layer 1, then node 2 after layer 2. Why does it take two layers to reach node 2, not one?

Receptive field grows by exactly one hop per layer.

Node 0’s neighbors are \(\{1, 3\}\) — one hop away. Node 2 is two hops away via \(0\to1\to2\) or \(0\to3\to2\). One layer reaches one hop. Two layers reach two hops. This is purely graph distance.

This is called the receptive field — directly analogous to CNN receptive field, where stacking convolutional layers expands the spatial region each output pixel “sees.”

Question: The portion says mean pooling is preferred over sum because it’s invariant to graph size. Can you think of a scenario where sum pooling would actually lose information that mean pooling preserves, or vice versa?

  • Graph A: 3 nodes, all with feature value 1 \(\to\) sum = 3; mean = 1
  • Graph B: 6 nodes, all with feature value 1 \(\to\) sum = 6; mean = 1

If energy scales with atom count, sum is the natural choice — bigger molecule, larger sum, larger prediction. Mean would wash that out.

QM9 uses mean specifically because the model handles variable-size molecules and needs stable training — scale invariance is prioritized over size-sensitivity. Neither is universally correct — it’s a design choice driven by the task.

The new concepts to focus are:

Concept Core idea
Receptive field depth = hops seen
Global pooling nodes → single graph vector
Permutation invariance addition commutes, so node ordering doesn’t matter

Phase 6 - Input Projection

Phase 6.1 - Why project at all?

You have raw node features \(\mathbf{X}\) going into the network. On QM9, each atom carries 11 chemically meaningful features — atom type, charge, etc. On \(C_4\), each node carries a scalar, so \(F=1\).

The problem: every MPNN layer operates in a uniform hidden space of dimension \(H=64\). You cannot feed a 1-dimensional or 11-dimensional vector directly into a layer expecting 64-dimensional inputs.

The fix is a single linear projection — one MLP layer with no hidden depth:

\[\mathbf{H}^{(0)} = \text{ReLU}(\mathbf{X}\mathbf{W}_\text{in})\]

Symbol Shape Meaning
\(\mathbf{X}\) \([N \times F]\) raw features
\(\mathbf{W}_\text{in}\) \([F \times H]\) learned projection
\(\mathbf{H}^{(0)}\) \([N \times H]\) projected features

On \(C_4\) with \(F=1\), \(H=2\) for simplicity:

\[\mathbf{W}_\text{in} = \begin{bmatrix} 1 & 1 \end{bmatrix}, \quad \mathbf{X} = \begin{bmatrix} 2 \\\\ 1 \\\\ 1 \\\\ 1 \end{bmatrix}\]

\[\mathbf{H}^{(0)} = \text{ReLU}\left(\begin{bmatrix}2 \\\\ 1 \\\\ 1 \\\\ 1\end{bmatrix}\begin{bmatrix}1&1\end{bmatrix}\right) = \text{ReLU}\left(\begin{bmatrix}2&2 \\\\ 1&1 \\\\ 1&1 \\\\ 1&1\end{bmatrix}\right) = \begin{bmatrix}2&2 \\\\ 1&1 \\\\ 1&1 \\\\ 1&1\end{bmatrix}\]

Every node is now living in \(\mathbb{R}^2\) — same space, ready for message passing.

Correct. The matrix multiplication requires:

\[[N \times F] \cdot [F \times H] = [N \times H]\]

so \(\mathbf{W}_\text{in}\) must be \([F \times H] = [11 \times 64]\) — inner dimensions must match.

Phase 7 - Stacked Message Passing & Receptive Field

Phase 7.1 - What stacking means

Every node now lives in a uniform \(H\)-dimensional space. A single MPNN layer — everything you built in Phases 2–4 — expands each node’s awareness by exactly one hop. Stacking \(L\) layers expands it by \(L\) hops.

Formally, each layer takes the current hidden state and produces the next:

\[\mathbf{H}^{(l+1)} = \text{MPNNLayer}\left(\mathbf{H}^{(l)},\ \mathcal{E}\right)\]

The superscript tracks depth, not time. \(\mathbf{H}^{(0)}\) is the projection output. \(\mathbf{H}^{(1)}\) encodes 1-hop neighborhoods. \(\mathbf{H}^{(2)}\) encodes 2-hop neighborhoods. And so on.

Phase 7.2 - Receptive Field on \(C_4\)

Tracing node 0 specifically across \(L=3\) layers:

Layer 0 → 1: Node 0 aggregates from \(\{1, 3\}\). Its new state \(\mathbf{h}_0^{(1)}\) encodes itself plus its direct neighbors.

Layer 1 → 2: Now \(\mathbf{h}_1^{(1)}\) already encodes node 1’s neighborhood \(\{0,2\}\). When node 0 aggregates from node 1 at layer 2, it indirectly pulls in node 2’s information. Node 0 has now seen 2 hops.

Layer 2 → 3: \(\mathbf{h}_2^{(2)}\) already encodes node 2’s neighborhood \(\{1,3\}\), which itself encoded \(\{0,2\}\). Node 0 now sees the full graph.

The receptive field table makes this concrete:

Layer Node 0 sees
0 \(\{0\}\)
1 \(\{0,1,3\}\)
2 \(\{0,1,2,3\}\)
3 full graph

\(C_4\) has diameter 2 — any node reaches any other in at most 2 hops — so \(L=2\) already saturates the receptive field here. Layer 3 adds no new nodes but re-encodes richer higher-order structure.

Phase 7.3 - The CNN Analogy

This is exactly your CNN receptive field intuition:

CNN GNN
Stacking conv layers Stacking MPNN layers
Spatial neighborhood grows Graph neighborhood grows
Kernel size controls hop width Aggregation controls hop width
Deep = sees more pixels Deep = sees more nodes

The difference: CNN neighborhoods are regular grids. GNN neighborhoods are irregular graphs — each node can have a different number of neighbors.


Question: On a path graph \(0\leftrightarrow1\leftrightarrow2\leftrightarrow3\leftrightarrow4\) with \(L=2\) layers, which nodes can node 0 see?

The node 0 can see: \(\{0,1,2\}\).

Correct — two hops from node 0 on a path graph reaches nodes \(\{0,1,2\}\). Node 3 is 3 hops away, node 4 is 4 hops — both out of reach with \(L=2\).

Phase 8 - Global Mean Pooling

Phase 8.1 - The problem it solves

After \(L\) layers, you have \(\mathbf{H}^{(L)} \in [N \times H]\) — one rich vector per node. But the task is graph-level regression: predict a single number \(U_0\) for the entire molecule.

You need to compress \(N\) node vectors into one graph vector. This is pooling — same concept as spatial pooling in CNN, but over graph nodes instead of pixel grids.

Phase 8.2 - Mean Pooling on \(C_4\)

\[\mathbf{g} = \frac{1}{4}\left(\mathbf{h}_0^{(L)} + \mathbf{h}_1^{(L)} + \mathbf{h}_2^{(L)} + \mathbf{h}_3^{(L)}\right)\]

Using our \(H=2\) updated states from Phase 4:

\[\mathbf{g} = \frac{1}{4}\left(\begin{bmatrix}2 \\\\ 2\end{bmatrix} + \begin{bmatrix}1 \\\\3 \end{bmatrix} + \begin{bmatrix}1 \\\\ 2\end{bmatrix} + \begin{bmatrix}1 \\\\ 3\end{bmatrix}\right) = \frac{1}{4}\begin{bmatrix}5 \\\\ 10\end{bmatrix} = \begin{bmatrix}1.25 \\\\ 2.5\end{bmatrix}\]

One vector summarizing the entire graph.

Phase 8.3 - The Batch Tensor

In practice, a training batch contains multiple graphs packed together. Nodes from different graphs sit in the same matrix but must pool separately.

The batch tensor handles this — it maps every node to its graph index:

\[\texttt{batch} = [0,0,0,1,1]\]

This says nodes \(\{0,1,2\}\) belong to graph 0, nodes \(\{3,4\}\) belong to graph 1. scatter_add_ then accumulates by graph index — exactly the same mechanic as Phase 0.4, but now the index is a graph label instead of a node label.

\[\mathbf{g}_0 = \frac{\mathbf{h}_0+\mathbf{h}_1+\mathbf{h}_2}{3}, \quad \mathbf{g}_1 = \frac{\mathbf{h}_3+\mathbf{h}_4}{2}\]

Two graphs, two pooled vectors, computed in one pass.


Question: If a batch contains three graphs with 4, 2, and 6 nodes respectively, what does batch look like and what shape is the pooled output?

One entry per node, grouped by graph, one pooled vector per graph, i.e.:

batch \(= [0,0,0,0,1,1,2,2,2,2,2,2]\), output \([3 \times H]\)

Phase 9 - Readout MLP

Phase 9.1 - What it does

You now have a single graph vector \(\mathbf{g} \in \mathbb{R}^H\) summarizing the entire molecule.

You have \(\mathbf{g} \in \mathbb{R}^{64}\) — a rich graph-level summary. The final step compresses it to a single scalar \(\hat{y}\):

\[\hat{y} = \mathbf{W}_2\,\text{ReLU}(\mathbf{W}_1\mathbf{g} + \mathbf{b}_1) + \mathbf{b}_2\]

Two MLP layers — nothing new mathematically. Just the same affine-then-nonlinearity chain from Phase 0.3, applied twice.

Tracing shapes with \(H=2\) and an intermediate dimension of 2:

Step Operation Shape
Input \(\mathbf{g}\) \([2]\)
Layer 1 \(\mathbf{W}_1\mathbf{g}+\mathbf{b}_1\) \([2\times2]\cdot[2]=[2]\)
ReLU elementwise \([2]\)
Layer 2 \(\mathbf{W}_2(\cdot)+\mathbf{b}_2\) \([1\times2]\cdot[2]=[1]\)
Output \(\hat{y}\) scalar

Phase 9.2 - Permutation Invariance

This is the final architectural guarantee worth understanding precisely.

Suppose you relabel \(C_4\)’s nodes — swap node 0 and node 2. Does \(\hat{y}\) change?

It doesn’t — and here’s exactly why, tracing back through the pipeline:

  • Mean pooling sums node vectors and divides. Addition commutes — order of summation is irrelevant. \(\mathbf{g}\) is identical regardless of node ordering.
  • MPNN layers aggregate via scatter_add_ — also a sum, also commutative.
  • Readout MLP is deterministic given \(\mathbf{g}\) — same input, same output.

The invariance is structural, not learned. No amount of training could break it — it is baked into the choice of sum/mean as the aggregation operator.

This matters physically: a molecule’s energy \(U_0\) is a property of its atomic structure, not of how you number its atoms.

Phase 10 - Full Pipeline on \(C_4\)

Putting everything together in one trace:

\[\underbrace{\begin{bmatrix}2 \\\\ 1 \\\\ 1 \\\\ 1\end{bmatrix}}_{\mathbf{X}\ [4\times1]} \xrightarrow{\mathbf{W}_\text{in}} \underbrace{\begin{bmatrix}2&2 \\\\ 1&1 \\\\ 1&1 \\\\ 1&1\end{bmatrix}}_{\mathbf{H}^{(0)}\ [4\times2]} \xrightarrow{L\times\text{MPNN}} \underbrace{\begin{bmatrix}2&2 \\\\ 1&3 \\\\ 1&2 \\\\ 1&3\end{bmatrix}}_{\mathbf{H}^{(L)}\ [4\times2]} \xrightarrow{\text{mean}} \underbrace{\begin{bmatrix}1.25 \\\\ 2.5\end{bmatrix}}_{\mathbf{g}\ [2]} \xrightarrow{\text{MLP}} \underbrace{\hat{y}}_{\text{scalar}}\]

Every shape transition follows one rule: node-indexed \([N\times H]\) or edge-indexed \([2E\times H]\), collapsing to a single vector at pooling.
